# ТЗ: перенос сайта на Django с БД заявок и развёртыванием

## 1. Цели

- Перенести текущий статический сайт (HTML/CSS/JS, формы обратной связи) на **Django**.
- Хранить заявки (лиды) из всех форм в **базе данных** с возможностью просмотра в админке.
- Обеспечить **простое развёртывание** на сервере (VPS, облако или хостинг с поддержкой Python).

---

## 2. Текущее состояние

- **Структура:** статические HTML-страницы в корне; общий header и footer должны быть статическими (в Django — подключаться в шаблонах через `{% include %}`, без подгрузки через JS).
- **Стили и скрипты:** `css/styles.css`, `js/header.js`, `js/main.js`.
- **Формы:** формы с атрибутом `data-lead` (главная, контакты, продукт и т.д.) отправляют POST на `/api/leads`; сейчас есть простой Express-сервер, который пишет в `leads.json`.
- **Контент:** многостраничный сайт (главная, решение, техника, сценарии, кейсы, блог, демо, FAQ, контакты, о компании, партнёры, материалы, политики и т.д.).

---

## 3. Требования к реализации

### 3.1. Фреймворк и окружение

- **Django** (актуальная LTS-версия, например 5.x).
- **Python** 3.10+.
- **База данных:** SQLite для разработки; для продакшена — **PostgreSQL** (или SQLite, если хостинг без БД).
- **Статика:** собирать в `STATIC_ROOT`, отдавать через `whitenoise` или веб-сервер (Nginx).
- **Медиа (опционально):** загрузки файлов в `MEDIA_ROOT` при необходимости.

### 3.2. Структура проекта

Предлагаемая структура:

```
project/
├── manage.py
├── requirements.txt
├── .env.example
├── config/                 # настройки Django
│   ├── settings/
│   │   ├── base.py
│   │   ├── dev.py
│   │   └── prod.py
│   ├── urls.py
│   └── wsgi.py
├── apps/
│   ├── pages/              # статические страницы (шаблоны)
│   │   ├── views.py
│   │   ├── urls.py
│   │   └── templates/pages/
│   └── leads/              # заявки
│       ├── models.py
│       ├── admin.py
│       ├── views.py
│       ├── urls.py
│       └── forms.py
├── static/                 # CSS, JS, собранная статика
├── templates/              # базовый layout, header, footer
│   ├── base.html
│   ├── partials/
│   └── ...
└── media/                  # при необходимости
```

- Текущие HTML-страницы переносятся в **шаблоны Django** (можно один каталог `templates/pages/` с подкаталогами или плоский список `index.html`, `solution.html` и т.д.).
- Header и footer выносятся в **include**-шаблоны (`{% include 'partials/header.html' %}`), без `fetch`; при необходимости общая разметка в `base.html`.

### 3.3. База данных и модель заявок

- **Приложение `leads`:**
  - Модель **Lead** (или **LeadRequest**):
    - `source` — источник формы (например `main`, `contact-demo`, `product-question`); CharField.
    - `payload` — JSON с полями формы (имя, email, телефон, компания, сообщение и т.д.); JSONField или TextField с JSON.
    - `created_at` — дата/время создания; DateTimeField (auto_now_add=True).
    - Опционально: `email`, `phone` — для удобного поиска в админке.
  - Регистрация модели в **Django Admin**: список заявок, фильтр по источнику и дате, поиск по email/телефону.

- **API приёма заявок:**
  - Один endpoint (например `POST /api/leads/` или `POST /leads/submit/`), принимающий JSON как сейчас: `{ "source", "data": { ... } }`.
  - В представлении: валидация (например обязательные поля в зависимости от source), сохранение в модель Lead, ответ `{ "ok": true }` или коды ошибок.
  - **CSRF:** для AJAX с других доменов рассмотреть CORS и/или освобождение endpoint от CSRF по токену (по необходимости); для отправки с того же домена — использовать cookie CSRF Django.

### 3.4. Формы на фронте

- Текущий `main.js` оставить логику отправки (сбор полей формы, `fetch`), изменить только URL запроса на новый Django endpoint (например `/api/leads/` или `/leads/submit/`).
- Успех/ошибка: как сейчас — показ блока «Заявка отправлена» и скрытие полей формы.

### 3.5. Развёртывание

- **Требования:** возможность запуска Python, установки пакетов (pip), при продакшене — PostgreSQL (или оставить SQLite для минимального хостинга).
- **Документация в репозитории:**
  - **README** с шагами:
    - клонирование репо, создание venv, `pip install -r requirements.txt`;
    - копирование `.env.example` в `.env`, настройка `SECRET_KEY`, `DEBUG`, `ALLOWED_HOSTS`, при необходимости `DATABASE_URL`;
    - `python manage.py migrate`, `python manage.py createsuperuser`;
    - запуск: `python manage.py runserver` (dev) или через **gunicorn** + **Nginx** (prod).
  - Краткая инструкция по деплою на типовой VPS (Ubuntu): Nginx, gunicorn, systemd unit, статика (`collectstatic`), пример конфига Nginx.
- **Переменные окружения:** все секреты и настройки окружения (SECRET_KEY, DB, DEBUG, ALLOWED_HOSTS) — из `.env`, не в коде.

### 3.6. Дополнительно

- **Язык и локали:** сайт на русском; при необходимости заложить использование `i18n` Django.
- **Логирование:** логировать ошибки при сохранении заявки (в файл или stderr), не отдавать детали пользователю.
- **Миграции:** все изменения моделей — только через миграции Django; миграции хранить в репозитории.

## 4. Этапы работ (кратко)

1. Инициализация проекта Django, приложения `pages` и `leads`, настройка настроек (base/dev/prod), `requirements.txt`, `.env.example`.
2. Модель Lead, миграции, регистрация в админке.
3. Endpoint приёма заявок (POST), валидация, сохранение в БД.
4. Перенос HTML-страниц в шаблоны (base, header, footer, страницы), подключение CSS/JS.
5. Обновление JS формы: URL на новый endpoint, при необходимости заголовок CSRF.
6. Инструкции по запуску и деплою в README + при необходимости отдельный файл DEPLOY.md.
7. Проверка отправки заявок и отображения в админке; проверка развёртывания на тестовом сервере.

---

## 5. Критерии приёмки

- Сайт открывается по всем текущим URL (главная, решение, техника, контакты и т.д.), внешний вид и поведение соответствуют текущему сайту.
- Отправка формы с любой страницы сохраняет заявку в БД и показывает сообщение об успехе.
- В Django Admin виден список заявок с полями (source, payload, created_at и при необходимости email/phone), есть фильтры и поиск.
- По README (и при необходимости DEPLOY) можно развернуть проект на чистом сервере с Python и PostgreSQL (или SQLite) и запустить сайт через gunicorn + Nginx.

---

## 6. Файлы для приложения в репозитории

- `requirements.txt` с версиями: Django, gunicorn, whitenoise, python-dotenv, psycopg2-binary (для PostgreSQL).
- `.env.example` с переменными: `SECRET_KEY`, `DEBUG`, `ALLOWED_HOSTS`, `DATABASE_URL` (опционально).
- Конфиг Nginx (пример) и unit systemd для gunicorn можно положить в `deploy/` или в корень с пометкой в README.

Если нужно, могу следующим шагом набросать структуру Django-проекта (команды `startproject`/`startapp`, примеры `settings`, `urls`, модель Lead и view для API).
